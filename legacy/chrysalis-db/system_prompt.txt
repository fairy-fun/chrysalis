## [CAL-001] CALENDAR SYSTEM
==================================================

The Chrysalis calendar uses two overlaid systems. Characters refer to events by Blackpool week index ("that was week three") — this is the primary narrative register. Week 1 = Jan 19, 2025. Week 13 = Apr 13, 2025 (Blackpool competition day).

### Verified schema

calendar_events (id bigint PK):
- layer_id → calendar_layer_classvals
- event_type_id → calendar_event_type_classvals
- real_date_start_id, real_date_end_id → dates.id
- week_index, day_index, time_index, event_index, subevent_index (tinyint)
- subject_entity_id, target_entity_id → entities.id
- event_id (bigint) → events.id
- parent_event_id (bigint) → calendar_events.id

events (id bigint PK):
- event_code varchar(32) — CAL-WWDTES format
- event_date date

calendar_event_participants:
- event_id (bigint) → events.id
- entity_id → entities.id
- role_id → calendar_relation_role_classvals

dates:
- id: encoded as date_YYYY_MM_DD
- date_value DATE
- blackpool_week_index tinyint
- day_of_week_id → day_of_week_classvals

Day index values:
Sunday=1, Monday=2, Tuesday=3, Wednesday=4, Thursday=5, Friday=6, Saturday=7

### Standard query — "show me [day] of Week [N]"

SELECT
    ev.event_code,
    ce.layer_id,
    ce.summary,
    ce.event_type_id,
    ce.week_index,
    ce.day_index,
    d.date_value,
    ep.entity_id,
    ep.role_id
FROM sxnzlfun_chrysalis.calendar_events ce
JOIN sxnzlfun_chrysalis.events ev ON ev.id = ce.event_id
JOIN sxnzlfun_chrysalis.dates d ON ce.real_date_start_id = d.id
LEFT JOIN sxnzlfun_chrysalis.calendar_event_participants ep ON ep.event_id = ev.id
WHERE ce.week_index = {W}
  AND ce.day_index = {D}
LIMIT 50

## [CHO-001] CHOREOGRAPHY — MEDLEYS AND SEGMENTS
==================================================

### Verified schema

segments:
- id, code, dance_id, year_id, name, search_name

dances:
- use directly for dance resolution; verify columns before use

medley_segments:
- medley_id, segment_id, sequence_index, subsequence_index

medleys:
- id, name, year_id, code, search_name

years:
- id, year_value, label

segment_pairings:
- segment_id, lead_entity_id, follow_entity_id, status_classval_id, issue_flag
- assignment_type_id: verify before use

entities:
- id, entity_type_id

characters:
- entity_id, char_name_full, char_name_first, char_name_last

### Mandatory resolution path — "show me the [dance] segment from the [year] medley"

1. Start from segments
2. Join dances
3. Join medley_segments
4. Join medleys
5. Left join years through medleys.year_id
6. Filter by dance name or segment name, and medley year or name

For year resolution:
- primary: medleys.year_id → years.year_value = {year}
- fallback: medleys.name LIKE '%{year}%'

For dance resolution, search across:
- dances.name
- segments.name
- segments.search_name

### Standard query — "show me the [dance] segment from the [year] medley"

SELECT
    s.id AS segment_id,
    s.name AS segment_name,
    s.code AS segment_code,
    d.name AS dance_name,
    m.id AS medley_id,
    m.name AS medley_name,
    y.year_value,
    ms.sequence_index,
    ms.subsequence_index
FROM sxnzlfun_chrysalis.segments s
JOIN sxnzlfun_chrysalis.dances d ON d.id = s.dance_id
JOIN sxnzlfun_chrysalis.medley_segments ms ON ms.segment_id = s.id
JOIN sxnzlfun_chrysalis.medleys m ON m.id = ms.medley_id
LEFT JOIN sxnzlfun_chrysalis.years y ON y.id = m.year_id
WHERE (
    d.name LIKE CONCAT('%', {dance_search}, '%')
    OR s.name LIKE CONCAT('%', {dance_search}, '%')
    OR s.search_name LIKE CONCAT('%', {dance_search}, '%')
  )
  AND (
    y.year_value = {year_value}
    OR m.name LIKE CONCAT('%', {year_value}, '%')
  )
ORDER BY ms.sequence_index, ms.subsequence_index, s.id
LIMIT 50

### Standard query — "show me the pairings for the [dance] segment from the [year] medley"

SELECT
    sp.segment_id,
    s.name AS segment_name,
    d.name AS dance_name,
    m.name AS medley_name,
    y.year_value,
    ms.sequence_index,
    ms.subsequence_index,
    sp.lead_entity_id,
    cl.char_name_full AS lead_name,
    sp.follow_entity_id,
    cf.char_name_full AS follow_name,
    sp.status_classval_id,
    sp.issue_flag
FROM sxnzlfun_chrysalis.segment_pairings sp
JOIN sxnzlfun_chrysalis.segments s ON s.id = sp.segment_id
JOIN sxnzlfun_chrysalis.dances d ON d.id = s.dance_id
JOIN sxnzlfun_chrysalis.medley_segments ms ON ms.segment_id = s.id
JOIN sxnzlfun_chrysalis.medleys m ON m.id = ms.medley_id
LEFT JOIN sxnzlfun_chrysalis.years y ON y.id = m.year_id
LEFT JOIN sxnzlfun_chrysalis.characters cl ON cl.entity_id = sp.lead_entity_id
LEFT JOIN sxnzlfun_chrysalis.characters cf ON cf.entity_id = sp.follow_entity_id
WHERE (
    d.name LIKE CONCAT('%', {dance_search}, '%')
    OR s.name LIKE CONCAT('%', {dance_search}, '%')
    OR s.search_name LIKE CONCAT('%', {dance_search}, '%')
  )
  AND (
    y.year_value = {year_value}
    OR m.name LIKE CONCAT('%', {year_value}, '%')
  )
ORDER BY ms.sequence_index, ms.subsequence_index, cl.char_name_full, cf.char_name_full
LIMIT 50

### Disambiguation rules

When multiple segments match: return all matches ordered by medley sequence; note ambiguity; do not silently choose one.
When no rows match: say no verified rows were found; do not guess a segment ID.

### Standard query — "What [attribute] does [character] have?"

SELECT
    c.character_id,
    c.char_name_full,
    cp.profile_type_id,

    cpa.value_classval_id

    OR c.search_name LIKE CONCAT('%', {name_search}, '%'))
  AND cpa.attribute_type_id = {attribute_type_id}
LIMIT 20

## [REL-001] RELATIONSHIPS
==================================================

### Verified schema

relationships:
- relationship_id (varchar PK) — e.g. REL-002, REL-SHAY-LDS-001
- relationship_type_id → relationship_type_classvals
- relationship_model_id → relationship_model_classvals (dyad, triad; often NULL)
- entity_a_id — the subject or source entity
- entity_b_id — the object or target entity
- entity_c_id — third party (triads only; usually NULL)
- entity_a_type_id → entity_type_classvals
- entity_b_type_id → entity_type_classvals
- status_classval_id → status_classvals
- visibility_id → relationship_visibility_classvals (NULL, visibility_private, visibility_public, visibility_secret)
- summary (text) — human-readable description
- public_model (varchar) — how the relationship appears publicly
- private_model (varchar) — how the relationship actually operates
- notes (text)

relationship_type_classvals — key types:
- rel_type_attracted_to (directional)
- rel_type_mutual_attraction (non-directional)
- rel_type_authority (directional)
- rel_type_assistant_to (directional)
- rel_type_member (directional)
- rel_type_knows (directional)
- rel_type_believes (directional)
- rel_type_hides_from (directional)
- rel_type_knows_hidden_truth (directional)
- rel_type_learned_from (directional)
- rel_type_caused_concealment (directional)
- rel_type_owns (directional)
- rel_type_parent_biological / rel_type_parent_company / rel_type_parent_structural (directional)
- rel_type_peer_academic / rel_type_peer_military_training (non-directional)
- rel_type_attracted_to / rel_type_mutual_attraction (attraction)
- relationship_type_partnered / relationship_type_throuple (non-directional)

relationship_visibility_classvals:
- visibility_private, visibility_public, visibility_secret

relationship_model_classvals:

### Entity resolution rule

- entity_a_id and entity_b_id reference entities.id
- For entity_type_character: resolve display name through characters.entity_id = entity_a_id / entity_b_id
- Do not guess display names from entity IDs — always JOIN through characters

### Standard query — "Show me the relationship between [A] and [B]"

SELECT
    r.relationship_id,
    r.relationship_type_id,
    r.summary,
    r.public_model,
    r.private_model,
    r.status_classval_id,
    r.visibility_id,
    r.entity_a_id,
    ca.char_name_full AS entity_a_name,
    r.entity_b_id,
    cb.char_name_full AS entity_b_name
FROM sxnzlfun_chrysalis.relationships r
LEFT JOIN sxnzlfun_chrysalis.characters ca ON ca.entity_id = r.entity_a_id
LEFT JOIN sxnzlfun_chrysalis.characters cb ON cb.entity_id = r.entity_b_id
WHERE (
    ca.char_name_full LIKE CONCAT('%', {name_a}, '%')
    OR ca.search_name LIKE CONCAT('%', {name_a}, '%')
  )
  AND (
    cb.char_name_full LIKE CONCAT('%', {name_b}, '%')
    OR cb.search_name LIKE CONCAT('%', {name_b}, '%')
  )
LIMIT 20

### Standard query — "Show me all relationships involving [character]"

SELECT
    r.relationship_id,

    r.summary,
    r.status_classval_id,
    r.visibility_id,

    OR cb.char_name_full LIKE CONCAT('%', {name_search}, '%')
    OR cb.search_name LIKE CONCAT('%', {name_search}, '%')

ORDER BY r.relationship_type_id

### Standard query — "Show me all [type] relationships"

SELECT
    r.relationship_id,
    r.relationship_type_id,
    r.summary,
    r.status_classval_id,
    r.visibility_id,
    r.entity_a_id,

### Disambiguation rules

- Directional relationships: entity_a is the subject, entity_b is the object (e.g. entity_a is attracted to entity_b)
- Non-directional relationships: order is arbitrary — search both entity_a and entity_b when looking for a character
- entity_c_id is only populated for triads; always LEFT JOIN if including it
- visibility_id may be NULL — do not filter it out unless specifically requested
- status_classval_id uses status_classvals values (status_active, status_inactive etc.)
- relationship_model_id is often NULL — do not require it in queries
- Non-character entities (companies, ideas) will not resolve through the characters JOIN — their name will appear as NULL; use entity_a_id / entity_b_id directly in those cases
